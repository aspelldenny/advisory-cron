### Assumptions trong phiếu — SAI so với code thật:

- **Anchor #19** (Linux `/usr/bin/crontab` present): N/A for P012 (Linux module is stub `bail!`) — correctly deferred to P013 verification. Not a mismatch.
- **Clippy: `PlatformScheduler::default()` on unit struct**: Phiếu's Task 9 example used `PlatformScheduler::default()`. Clippy `default_constructed_unit_structs` lint rejects `::default()` on unit structs — all 6 instantiation sites changed to `PlatformScheduler` (bare struct expression). Tầng 2 self-resolved.
- **`cargo fmt` ordering**: `mod scheduler;` alphabetically falls after `mod runner;` not before `mod mcp;`. rustfmt reordered to alphabetical. 3 files reformatted. Tầng 2 self-resolved.
- **macos module gating**: Phiếu Task 1 proposed `pub mod macos;` (unconditional). Without `#[cfg(target_os = "macos")]` gate, Linux compile produces 14 dead-code warnings for all macos.rs internals. Added `#[cfg(target_os = "macos")]` to `pub mod macos;` declaration in `scheduler/mod.rs` to suppress warnings on Linux. Tầng 2 self-resolved.
- **NoopScheduler / RegisterIntent dead-code warnings on Linux**: Both structs are `pub` but only constructed on macOS targets. Added `#[allow(dead_code)]` to suppress on Linux. Tầng 2 self-resolved.

### Edge cases / limitations phát hiện thêm:

- **P013 watch-item (noted in CHALLENGE V1, DP4):** `RegisterOutput.plist_path: PathBuf` is populated via `report.plist_path.unwrap_or_default()`. On Linux, `CrontabScheduler::register` currently bails before returning — so `plist_path` is never empty on Linux in P012. BUT when P013 implements real `CrontabScheduler::register` and returns `RegisterReport { plist_path: None }`, the CLI render `cli/register.rs:51` will print `"  plist: "` with a blank path. P013 must either (a) populate `plist_path` with a crontab-equivalent descriptor or (b) gate the CLI render path on `!plist_path.as_os_str().is_empty()`. Logged here for Architect's P013 phiếu acceptance criteria.
- **Test count delta (Linux vs macOS):** 11 lib tests from `scheduler::macos::tests` are now `#[cfg(target_os = "macos")]` gated — they do NOT run on Linux. 3 new `scheduler::linux::tests` run on Linux (+3). 4 integration tests in `tests/cli_register.rs` gated `#[cfg(target_os = "macos")]` (write plist + launchctl cleanup). Net Linux test count: 129 (vs macOS: expect ~144+3=147 after P012 lands on macOS CI).
- **`CrontabScheduler::status()` error swallowed by `core::status::run`:** The error handler in `core/status.rs` treats any `Err` from `scheduler.status()` as `is_registered: false` (silent fallback). On Linux in P012, this means `advisory-cron status` exits 0 with "not loaded" output instead of exit 3 with "P013 chưa ship" error. This is correct for P012 (status is read-only, not an error state) but P013 may want to bubble the error through. Logged for P013 awareness.
