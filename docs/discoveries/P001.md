### Assumptions trong phiếu — SAI so với code thật:

- **Anchor #5 / tokio flavor:** Phiếu spec dùng `#[tokio::main]` (bare, defaults to `multi_thread` flavor), nhưng `Cargo.toml` chỉ có tokio features `["rt", "macros", "process", "time", "fs"]` — thiếu `rt-multi-thread`. `cargo check` fail với lỗi "The default runtime flavor is `multi_thread`, but the `rt-multi-thread` feature is disabled."

  **Sửa (Tầng 2 — internal function call shape, không ảnh hưởng CLI surface):** Đổi sang `#[tokio::main(flavor = "current_thread")]`. Behavior identical với stub CLI (no concurrent tasks). Zero dep change.

  **Lý do đây là Tầng 2:** Không thay đổi CLI surface, không thay đổi module layout, không thay đổi `Cargo.toml`. Là internal detail của `main()` function annotation.

  **Ghi chú cho Phase 1.4 (runner.rs):** Khi `runner.rs` cần spawn concurrent `tokio::process::Command`, Architect cần đánh giá lại: (a) giữ `current_thread` + `FuturesUnordered` (ok cho 1 task/fire), hoặc (b) bump tokio feature to include `rt-multi-thread` (Tầng 1 phiếu 1.4). Hiện tại stub không spawn bất cứ gì, `current_thread` là đúng.

### Edge cases / limitations phát hiện thêm:

- **Clap exit 2 vs spec exit 2:** Khi user gõ `advisory-cron bogus-subcommand`, clap trả exit 2 ("error: unrecognized subcommand"). Spec exit 2 = "Config not found / invalid". Hai context khác nhau: clap parse error precedes any app logic. Không collision trong Phase 1.1 (stubs không load config). Phiếu đã note risk; Architect defer handle đến Phase 1.2 khi `init` có config logic thật — không cần action bây giờ.

- **`cargo update --dry-run` shows `displaydoc v0.2.5 -> v0.2.6` available** (transitive dep của clap). P001 không run `cargo update`, `Cargo.lock` stays pinned. Minor bump, không breaking. Phiếu đã note trong Challenge Turn 1. Ghi lại để Phase 1.2 awareness.

- **Bin name với hyphen:** `env!("CARGO_BIN_EXE_advisory-cron")` — Cargo bin EXE env var dùng tên binary đúng với hyphen. Integration test compile và run clean. Không issue.
